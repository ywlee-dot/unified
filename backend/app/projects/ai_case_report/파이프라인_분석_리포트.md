## Step 5: Group by 실적

### n8n 노드
- 타입: `n8n-nodes-base.code` (JavaScript)

### 코드 로직
```javascript
const items = $input.all();
const groups = {};

for (const item of items) {
  const meta = item.json.메타데이터 || {};
  const 실적번호 = meta.실적번호;
  const key = String(실적번호);

  if (!groups[key]) {
    groups[key] = {
      실적번호, 파일수: 0, 파일목록: [], 문서목록: []
    };
  }
  groups[key].파일목록.push(meta.원본파일명);
  groups[key].문서목록.push(item.json);
  groups[key].파일수 = groups[key].문서목록.length;
}

return Object.values(groups).sort((a,b) => a.실적인덱스 - b.실적인덱스);
```

### 수행 내용
- 같은 실적번호의 파일들을 하나의 그룹으로 합침
- 실제 서비스에서는 실적 1에 여러 파일을 업로드할 수 있으므로 그루핑이 필요
- 이번 시뮬레이션에서는 1실적=1파일이므로 4그룹 생성

## Step 6: Prepare AI Review1 + Label Report Items1

### n8n 노드
- 타입: `n8n-nodes-base.code` (JavaScript) x 2

### Prepare AI Review1 코드
```javascript
return items.map(item => {
  const 문서목록 = item.json.문서목록 || [];
  const 검토대상 = 문서목록.map((doc, i) => {
    const 파일명 = doc.메타데이터?.원본파일명 || `파일${i+1}`;
    return `[문서 ${i+1}: ${파일명}]\n${doc.원본텍스트 || ''}`;
  }).join('\n\n---\n\n');

  return { json: { ...item.json, 검토대상: 검토대상.trim() } };
});
```

### Label Report Items1 코드
```javascript
// 실적 내 전체 원본텍스트 합산 (AI가 직접 읽고 판단)
const 전체원본텍스트 = 문서목록.map((doc, i) => {
  const 파일명 = doc.메타데이터?.원본파일명 || `파일${i+1}`;
  return `[문서 ${i+1}: ${파일명}]\n${doc.원본텍스트 || ''}`;
}).join('\n\n---\n\n');

return { json: { 실적번호, 파일목록, 전체원본텍스트 } };
```

### 수행 내용
- 그룹 내 여러 문서의 원본텍스트를 `---` 구분자로 합산
- AI가 한 번에 읽을 수 있도록 전체 텍스트 + 파일 목록 준비

## Step 7: Build AI Prompt

### n8n 노드
- 타입: `n8n-nodes-base.code` (JavaScript)

### 프롬프트 구조

```
원본 문서 내용 (최대 14,000자)
파일명 목록 (추진실적명 작성 참고용)
실행 기준일

항목별 작성 지시:
  1. 추진실적명 — 파일명에서 핵심 키워드 추출
  2. 추진기간 — 문서에서 기간 정보 찾기, 없으면 추론
  3. 추진배경 및 목적 — 있으면 정리, 없으면 추진내용에서 추론
     (번호 목록, 음슴체, 2~4개)
  4. 추진내용 — 요약, 번호+하이픈 구조, 음슴체
  5. 기대효과 — 내부 관점 중심, 2~3개, 음슴체
  6. 향후계획 — 있으면 정리, 없으면 추론, 2~3개, 음슴체

출력: 순수 JSON 1개 (마크다운/설명 금지)
```

- **사실 기반**: 문서에 없는 사실/수치/기관명 창작 금지
- **원본 텍스트 14,000자 제한**: 초과 시 잘림

## Step 8: AI Agent - 보고서 작성 및 검토

### n8n 노드
- 타입: `@n8n/n8n-nodes-langchain.agent`
- LLM: `Google Gemini Chat Model4` (models/gemini-pro-latest)
- 프롬프트: `{{ $json._ai_prompt }}` (Step 7에서 생성)

### 시스템 프롬프트
```
당신은 공공기관 AI 도입활용 사례 정성보고서 작성 전문가입니다.

역할:
- 제공된 원본 문서를 직접 읽고, 각 항목이 문서에 존재하는지 스스로 판단
- 존재하면: 해당 내용을 지정된 형식에 맞게 정리
- 없으면: 추진내용을 바탕으로 합리적으로 추론하여 작성

작성 원칙:
1. 반드시 제공된 원본 문서 내용에만 근거
2. 문서에 없는 사실, 수치, 기관명, 사업명을 절대 창작하지 않음
3. 각 항목의 형식 규칙을 반드시 준수
4. 응답은 반드시 순수 JSON 형식으로만 출력 (마크다운 금지)

형식 규칙:
- 추진배경및목적, 기대효과, 향후계획: 번호 목록 + 음슴체
- 추진내용: 번호. 항목명 → 하이픈(-)으로 세부내용
- 존댓말 절대 금지
```

4개 PDF의 원본 텍스트를 각각 읽고, 프롬프트의 6항목 지시에 따라 JSON 보고서 생성.

## Step 9: Parse AI Response

### n8n 노드
- 타입: `n8n-nodes-base.code` (JavaScript)

### 코드 로직
```javascript
function parseReportJson(aiOutput, 파일목록) {
  try {
    const cleaned = String(aiOutput)
      .replace(/```json\s*/gi, '')
      .replace(/```\s*/g, '')
      .trim();
    return JSON.parse(cleaned);
  } catch (e) {
    // 파싱 실패 시 원본 텍스트를 추진내용에 넣고 나머지는 빈 값
    return {
      추진실적명: 파일목록.join(', '),
      추진기간: '',
      추진배경및목적: '',
      추진내용: String(aiOutput),
      기대효과: '',
      향후계획: ''
    };
  }
}
```

### 수행 내용
1. AI 응답에서 마크다운 코드블록 제거
2. JSON.parse() 시도
3. 성공: 6항목 구조화된 보고서 객체
4. 실패: 파일명을 실적명으로, AI 원본 응답을 추진내용에 넣는 폴백

## 발견된 주요 이슈

### 1. PDF 텍스트 추출 품질

**문제:** PyPDF2 / n8n readPDF의 텍스트 추출 품질이 PDF에 따라 크게 다름.
- [1] AI 신기술 이용료: 5페이지, 3,918자 → 테이블/레이아웃이 포함된 문서라 일부 텍스트 누락 가능
- [2] ChatGPT 구독 보고: 1페이지, 879자 → 매우 짧은 문서. □ 마커 없어서 섹션 추출 실패

**영향:** 섹션 구조화 실패 시 AI에게 비구조화 텍스트가 전달되어 보고서 품질 저하 가능

### 2. □ 마커 의존적 구조화

**문제:** Structure PDF Data1 노드가 `□` 마커에 전적으로 의존.
- `□`가 없는 문서(예: 코레일유통 ChatGPT 구독 보고)는 섹션 추출이 완전히 실패
- 다른 마커 (■, ○, ●, 1., (1) 등)를 사용하는 문서는 미지원

**개선:** 다중 마커 지원, 또는 AI에게 구조화를 맡기는 방식

### 3. 파이프라인 B에 품질 검증 없음

**문제:** AI가 한 번에 작성한 보고서가 바로 최종 산출물로 출력.
- 음슴체 미준수, 항목 누락, 사실과 다른 내용 등을 검증하는 단계 없음
- 파이프라인 A에는 다단계 검증 루프(평가→품질→재작성→재검토)가 있으나, B에는 없음

**영향:** B로 생성한 보고서는 수동 검수 필수

### 4. 원본 텍스트 14,000자 제한

**문제:** Build AI Prompt에서 `전체원본텍스트.slice(0, 14000)` 처리.
- 이번 4개 PDF는 모두 14,000자 미만이라 문제 없음
- 하지만 대용량 PDF (보고서 50페이지 등)는 내용이 잘려서 불완전한 보고서 생성

### 5. 파이프라인 A/B 코드 중복

**문제:** 핵심 로직(PDF 읽기, 구조화, 프롬프트 생성)이 두 파이프라인에 복제.
- Structure PDF Data / Structure PDF Data1
- Prepare AI Review / Prepare AI Review1
- Label Report Items / Label Report Items1
- Generate HTML Report / Generate HTML Report1

한쪽 수정 시 다른 쪽 동기화 필요. 유지보수 부담.

### 6. 콜백 네트워크 문제 (파이프라인 A)

다른 프로젝트와 동일:
- 콜백 URL이 `http://backend:8000` (Docker 내부)
- n8n이 외부 서버에 있으면 콜백 불가
