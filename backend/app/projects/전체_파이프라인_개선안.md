## Part 1. 공통 이슈 (전체 프로젝트 해당)

### 1-1. 콜백 네트워크 단절 [치명]

**영향 프로젝트:** 전체 (Webhook 파이프라인 사용 시)

n8n 워크플로우의 콜백/파일 업로드 URL이 `http://backend:8000` (Docker 내부 호스트명)으로 설정되어 있음. n8n이 외부 서버(168.107.58.30)에 있으면 이 주소에 도달할 수 없어서, AI가 결과를 만들어도 백엔드로 전달이 안 됨.

**해결 방안:**
- n8n 워크플로우의 콜백 URL을 백엔드의 외부 접근 가능 주소로 변경
- 또는 로컬 n8n(Docker 내부)에서 실행하면 `backend:8000`이 정상 작동

### 1-2. 파이프라인 A/B 코드 중복 [중간]

**영향 프로젝트:** 우수사례 검색, AI 보고서, 공공데이터 활용도

세 프로젝트 모두 하나의 n8n 워크플로우에 2개 파이프라인(Webhook용 A + Form용 B)이 공존하고, 핵심 로직 노드가 접미사("1")로 복제되어 있음. 한쪽을 수정하면 다른 쪽도 동일하게 수정해야 함.

**해결 방안:**
- n8n의 Sub-workflow 기능으로 공통 로직을 별도 워크플로우로 분리
- 또는 A 파이프라인만 유지하고, n8n Form은 Webhook을 호출하는 방식으로 통합

### 1-3. AI 응답 파싱 실패 시 에러 처리 없음 [높음]

**영향 프로젝트:** 전체

AI 응답이 JSON으로 파싱 불가능할 때:
- 업무규칙: Step 8-1에서 `parse_ok=false` 경로에 연결 노드 없음 → 워크플로우 멈춤
- 보고서: Parse AI Response에서 catch 블록이 있지만, 결과가 빈 보고서 → 사용자에게 에러 알림 없음

**해결 방안:**
- 파싱 실패 경로에 에러 콜백 노드 연결 (status: "failed" + error_message 전송)
- 프론트엔드에서 빈 결과 감지 시 경고 표시

### 1-4. stuck 상태 방지 없음 [중간]

**영향 프로젝트:** 전체 (Webhook 파이프라인)

n8n이 콜백을 못 보내면 DB에서 영구 "running" 상태. 사용자는 끝없이 "실행 중" 화면을 봄.

**해결 방안:**
- 백엔드에 TTL 체크 크론잡 추가 (30분 경과 시 자동 "failed" 처리)
- 프론트엔드에서도 일정 시간 경과 시 타임아웃 처리

## Part 2. 업무규칙 자동 생성 개선안

### 현재 결과 진단
```
29개 규칙 생성, 18/43 테이블 커버(42%)
시간순서 일관성 16개(55%) 편중 — 동일 패턴(MOD >= REG) 반복
```

### 개선안 2-1: TMP/설문 테이블 필터링 [난이도: 낮음]

**문제:** 43개 테이블 중 TMP(임시 테이블 4개)와 ETCDM(설문 테이블 2개)이 전체 컬럼의 26%를 차지. AI가 이 테이블에도 규칙을 생성하려고 시도하여 자원 낭비.

**필터링 규칙:**
```
제외 조건:
1. 테이블명이 "TMP"로 시작 → 임시 테이블
2. 컬럼 50개 초과 + 업무키(BIZ_CD, SBJT_NO) 없음 → 설문/로그 테이블
```

**기대 효과:**
- 프롬프트 50,498자 → 약 30,000자 (40% 절감)
- AI가 의미 있는 37개 테이블에 규칙 집중

### 개선안 2-2: SQL 컬럼 존재 자동 검증 [난이도: 중간]

**문제:** AI가 생성한 SQL에 실제 존재하지 않는 컬럼이 사용될 수 있음. 현재는 또 다른 AI(검토 Agent)가 이를 잡아야 하는데, AI도 놓칠 수 있음.

**검증 로직:**
```
1. 분석SQL에서 컬럼명 추출 (정규식)
2. 해당 테이블의 실제 컬럼 목록과 대조
3. 없는 컬럼 사용 시 → 자동 부적합 + 정확한 사유 전달
```

**기대 효과:** AI 환각(hallucination)으로 생긴 가짜 컬럼 참조 100% 감지

### 개선안 2-3: 테이블별 규칙 후보 힌트 사전 분석 [난이도: 중간]

**문제:** AI에게 테이블 메타데이터만 통째로 던지면 가장 쉬운 패턴(MOD >= REG)만 반복 생성.

**감지 가능한 8가지 패턴:**
```
1. 날짜 쌍 (REG_DTTM/MOD_DTTM) → 시간순서 (이미 잘 잡힘, 제한 필요)
2. 시작/종료 쌍 (STT_DATE/END_DATE) → 시간순서
3. 자기참조 FK → 순환참조 검사
4. 상위/하위 코드 (UPR_ 접두사) → 논리관계
5. YN 플래그 + 연관 컬럼 → 논리관계 (Y이면 값 존재)
6. 상태코드 (STATUS_CD) → 조건부 필수
7. 건수/금액/비율 숫자 → 계산식 (>= 0)
8. 월/일 범위 숫자 → 계산식 (1~12, 1~31)
```

프롬프트에 "이 테이블은 순환참조 검사가 가능해"라고 힌트를 추가하면 AI가 다양한 규칙을 생성함.

**기대 효과:** 동일 패턴 55% → 30% 이하, 테이블 커버리지 42% → 80%+

### 개선안 2-4: 데이터타입-SQL 연산 호환성 체크 [난이도: 중간]

**문제:** AI가 생성한 SQL이 실제 Oracle DB에서 실행하면 에러날 수 있음.

**체크 규칙:**
```
1. VARCHAR2 날짜 컬럼을 DATE와 직접 비교 → TO_DATE 필요
2. CLOB 컬럼에 = 연산자 사용 → DBMS_LOB 필요
3. DBMS별 함수 차이 (Oracle NVL vs PostgreSQL COALESCE)
```

**기대 효과:** 생성된 SQL을 실제 DB에 바로 실행 가능

### 개선안 2-5: 규칙 다양성 강제 검증 [난이도: 낮음]

**문제:** 29개 중 16개(55%)가 동일 패턴.

**검증 규칙:**
```
1. 동일 패턴 40% 초과 → 경고, 초과분 삭제 후 재생성
2. 품질지표 한 종류 60% 초과 → 경고
3. 필터링 후 테이블 커버율 70% 미만 → 미커버 목록으로 추가 생성
4. 한 테이블 4개 이상 → 초과분 삭제
```

### 개선안 2-6: 수정 후 재검토 루프 [난이도: 중간]

**문제:** 부적합 규칙을 수정한 뒤 재검토 없이 바로 최종 산출물에 합침. 수정된 SQL이 여전히 틀릴 수 있음.

**해결:** 수정 → 재검토 루프 (최대 2회)

### 적용 로드맵

| Phase | 개선안 | 난이도 | 효과 |
|---|---|---|---|
| 1 (즉시) | 2-1 TMP 필터링 + 2-5 다양성 검증 | 낮음 | 프롬프트 40% 절감, 편중 감지 |
| 2 (품질) | 2-2 컬럼 존재 검증 + 2-4 타입 호환 | 중간 | SQL 오류 사전 방지 |
| 3 (근본) | 2-3 힌트 분석 + 2-6 재검토 루프 | 중간 | 다양성 + 정확도 근본 해결 |

## Part 3. 민간 활용 우수사례 검색 개선안

### 현재 결과 진단
```
SerpAPI 40건 수집 → 38건(중복 제거) → 0건(AI 필터링)
모든 기사가 기관 자체 소식이거나, 날짜 범위 밖이거나, 기관과 무관
```

### 개선안 3-1: 다중 검색 전략 [난이도: 중간]

**문제:** 단일 검색어로 구글 뉴스만 검색. "민간이 공단 데이터를 활용"한 기사는 이 방식으로 잘 안 잡힘.

**해결:**
```
검색 1: "{기관명}" "공공데이터" "활용" (구글 뉴스)
검색 2: "{기관명}" "API" "서비스" (구글 뉴스)
검색 3: "{기관명 데이터 브랜드}" (예: "NCS", "Q-Net") (구글 뉴스)
검색 4: "{기관명}" (공공데이터포털 활용사례 검색)
검색 5: "{기관명 데이터}" (KIPRIS 특허 검색)
```

**기대 효과:** 뉴스 외 채널(특허, 공공데이터포털)까지 커버하여 실질적 사례 발굴

### 개선안 3-2: 검색어 자동 확장 [난이도: 낮음]

**문제:** 사용자가 입력한 검색어를 그대로 사용. 기관 자체 소식이 상위 노출.

**해결:**
```
사용자 입력: "데이터, 활용, 서비스"
자동 확장:
  - "한국산업인력공단 데이터 활용 앱"
  - "한국산업인력공단 공공데이터 민간"
  - "NCS API 활용 서비스" (기관 데이터 브랜드 자동 추가)
```

### 개선안 3-3: 날짜 파싱 정규화 [난이도: 낮음]

**문제:** SerpAPI의 "1주 전", "2주 전" 같은 상대 날짜를 절대 날짜로 변환하지 않음.

**해결:**
```javascript
function normalizeDate(dateStr) {
  const now = new Date();
  if (dateStr.includes('주 전')) {
    const weeks = parseInt(dateStr);
    return new Date(now - weeks * 7 * 86400000);
  }
  if (dateStr.includes('일 전')) {
    const days = parseInt(dateStr);
    return new Date(now - days * 86400000);
  }
  // "2025. 10. 28." 형식 파싱
  const match = dateStr.match(/(\d{4})\.\s*(\d{1,2})\.\s*(\d{1,2})/);
  if (match) return new Date(match[1], match[2]-1, match[3]);
  return null;
}
```

### 개선안 3-4: 평가 기간 동적 설정 [난이도: 낮음]

**해결:** Form Trigger에 "평가 시작일", "평가 종료일" 필드 추가. 프롬프트에서 동적으로 참조.

### 개선안 3-5: SerpAPI 대체 검토 [난이도: 높음]

**문제:** 무료 월 100건, 1회 4건 사용 → 월 25회 제한.

**대안:**

## Part 4. AI 도입활용 사례 보고서 + 공공데이터 활용도 제고 공통 개선안

두 프로젝트는 파이프라인 구조가 거의 동일 (PDF → 텍스트 추출 → AI 보고서 → HTML). 이슈도 공통.

### 현재 결과 진단
```
AI 보고서: 4건 PDF → 4건 보고서 (6항목)
공공데이터: 6건 PDF → 3실적 보고서 (5항목)
두 프로젝트 모두 원본 텍스트 반영도가 낮고, AI 추론에 의존하는 경향
```

### 개선안 4-1: 다중 마커 섹션 추출 [난이도: 낮음]

**문제:** Structure PDF Data 노드가 `□` 마커에만 의존. □가 없는 문서는 섹션 추출이 완전 실패. ChatGPT 구독 보고서(879자)가 대표적 사례.

**해결:**
```javascript
// 다중 마커 지원
const markerPatterns = [
  /□\s*([^\n]+)/g,        // □ 마커 (현재)
  /■\s*([^\n]+)/g,        // ■ 마커
  /●\s*([^\n]+)/g,        // ● 마커
  /○\s*([^\n]+)/g,        // ○ 마커
  /\d+\.\s+([^\n]+)/g,    // 1. 2. 3. 번호 마커
  /\(\d+\)\s*([^\n]+)/g,  // (1) (2) (3) 괄호 번호
];

// 각 패턴으로 시도, 가장 많은 섹션을 추출한 패턴 사용
let bestSections = {};
let bestCount = 0;
for (const pattern of markerPatterns) {
  const sections = extractWithPattern(text, pattern);
  if (Object.keys(sections).length > bestCount) {
    bestSections = sections;
    bestCount = Object.keys(sections).length;
  }
}
```

**기대 효과:** □ 마커가 없는 문서에서도 섹션 추출 가능. 특히 간략한 보고서(1~2페이지)의 구조화 성공률 향상.

### 개선안 4-2: 원본 텍스트 충실 반영 검증 [난이도: 중간]

**문제:** AI가 원본 텍스트의 구체적 내용(수치, 일정, 기관명 등)을 충분히 반영하지 않고 일반적 서술로 대체하는 경향.

**해결 (프롬프트 강화):**
```
추가 지시:
- 원본에 구체적 수치(참여자 수, 금액, 건수)가 있으면 반드시 보고서에 포함할 것
- 원본에 구체적 날짜/일정이 있으면 반드시 포함할 것
- "~기대됨", "~예상됨" 같은 추상적 표현보다 원본 기반 구체적 서술 우선
```

**검증 로직 (Post-AI):**
```
1. 원본에서 숫자(정규식 \d+)를 추출
2. 보고서에 해당 숫자가 포함되어 있는지 확인
3. 원본에 있는 숫자가 보고서에 없으면 → 경고 + 재작성 요청
```

### 개선안 4-3: 파이프라인 B에 품질 검증 추가 [난이도: 중간]

**문제:** 파이프라인 A에는 다단계 검증 루프(평가→품질→재작성→재검토)가 있지만, B는 AI 한 번 호출로 끝. 보고서 품질 보장 안 됨.

**해결:**
```
현재 B: AI 작성 → 파싱 → HTML
개선 B: AI 작성 → 파싱 → 형식 검증 → (미달 시 재작성) → HTML
```

형식 검증 항목:
```
1. 음슴체 준수 여부 (~임, ~함 vs ~습니다, ~합니다)
2. 번호 목록 형식 여부 (줄글 금지)
3. 각 항목이 최소 글자수 이상인지 (빈 항목 방지)
4. 원본 텍스트의 핵심 키워드가 보고서에 포함되어 있는지
```

### 개선안 4-4: 원본 텍스트 길이 제한 개선 [난이도: 낮음]

**문제:** `전체원본텍스트.slice(0, 14000)` / `slice(0, 12000)` 하드코딩. 대용량 PDF는 잘림.

**해결:**
```
1단계: 14,000자 이하 → 전체 텍스트 사용
2단계: 14,000자 초과 → 섹션별 요약 후 전달
  - 각 섹션의 앞 500자만 추출
  - 또는 AI에게 먼저 요약을 시킨 뒤, 요약본으로 보고서 작성
```

### 개선안 4-5: PDF 텍스트 추출 품질 향상 [난이도: 중간]

**문제:** PyPDF2 / n8n readPDF가 표, 이미지 포함 PDF에서 텍스트를 제대로 추출하지 못함.

**해결:**
- n8n에서 `pdfplumber` 또는 `pdf-parse` 같은 고품질 PDF 파서 사용
- 또는 OCR 기반 추출 (이미지 PDF 대응)
- 실적 1(수요조사 결과보고서)이 7페이지인데 2,692자만 추출된 건 표/차트가 많아서 텍스트가 적게 나온 것으로 추정

### 적용 로드맵

| Phase | 개선안 | 난이도 | 효과 |
|---|---|---|---|
| 1 (즉시) | 4-1 다중 마커 + 4-4 텍스트 제한 개선 | 낮음 | 섹션 추출 성공률 향상 |
| 2 (품질) | 4-2 원본 반영 검증 + 4-3 품질 검증 | 중간 | 보고서 품질 보장 |
| 3 (장기) | 4-5 PDF 추출 품질 | 중간 | 표/이미지 PDF 대응 |
