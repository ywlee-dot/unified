## Step 0: Binary 데이터 확인 및 변환

### n8n 노드
- 타입: `n8n-nodes-base.code` (JavaScript)
- 위치: Webhook 직후

### 코드 로직
```javascript
const items = $input.all();
const item = items[0];

// Binary 데이터 존재 확인
if (!item.binary || Object.keys(item.binary).length === 0) {
  throw new Error('업로드된 파일이 없습니다.');
}

// 첫 번째 binary 필드를 표준 이름 'data'로 변환
const binaryFieldName = Object.keys(item.binary)[0];
return [{
  json: item.json,
  binary: { 'data': item.binary[binaryFieldName] }
}];
```

### 수행 내용
- Webhook으로 들어온 multipart 요청에서 binary(파일) 데이터를 추출
- 필드명이 'file', 'upload' 등 다양할 수 있으므로 첫 번째 binary 필드를 `data`로 통일
- 이후 노드에서 `$input.item.binary.data`로 일관되게 접근 가능

### 입출력
- 입력: `{ json: { tracking_id: "uuid" }, binary: { file: <엑셀 바이너리> } }`
- 출력: `{ json: { tracking_id: "uuid" }, binary: { data: <엑셀 바이너리> } }`

### 시뮬레이션 결과
- 파일: 컬럼정의서_한국학진흥사업단 홈페이지.xlsx (87,911 bytes)

## Step 3: AI Agent - 업무규칙 생성

### n8n 노드
- 타입: `@n8n/n8n-nodes-langchain.agent`
- LLM: `Google Gemini Chat Model1` (models/gemini-pro-latest)
- 온도/파라미터: 기본값 (options: {})

### 프롬프트 구조

#### 시스템 프롬프트 (핵심 규칙)
```
역할: 데이터 품질 관리 전문가
생성 개수: 전체 최대 30개, 테이블당 1~2개, 테이블 분산 필수
품질지표: 논리관계 일관성 | 시간순서 일관성 | 계산식 (3개 중 택1)

업무규칙명 형식 (절대 준수):
  1) "컬럼명" "품질지표명"
  2) "컬럼명"와/과 "컬럼명" "품질지표명"

절대 금지:
  - 필수값/누락 검사 (IS NULL 단독)
  - 연도 허용범위 (현재년도+5 등)
  - 형식/패턴 검증 (YYYYMMDD, REGEXP_LIKE 등)

계산식 허용 범위:
  - 퍼센트 0~100, 금액/수량 >= 0
  - 시작값 <= 종료값 (수치)
  - A = B + C (산술관계)

출력: JSON 배열만 (마크다운/코드블록 금지)
```

#### 사용자 프롬프트
```
다음 테이블(들)의 컬럼 정보를 분석하여 업무규칙을 생성해주세요.
...
## 테이블 정보/컬럼 정보
{{ $json.chatInput }}
```
→ Step 2에서 생성한 50,498자의 테이블/컬럼 메타데이터가 들어감

### 수행 내용 (Claude가 Gemini 대신 수행)
43개 테이블의 컬럼 메타데이터를 분석하여 아래 기준으로 규칙 생성:

1. **시간순서 일관성** (16개): 거의 모든 테이블에 REG_DTTM(등록일시), MOD_DTTM(수정일시) 패턴이 존재. "수정일 >= 등록일" 규칙 적용. 일부 테이블은 시작일/종료일 비교.
2. **논리관계 일관성** (7개): 자기참조 FK(순환참조 방지), 상태코드-데이터 정합성, 년도 간 논리적 관계.
3. **계산식** (6개): 월(1~12), 건수(>=0), 조회수(>=0) 등 값 범위 검증.

### 입출력
- 입력: 시스템 프롬프트 + 50,498자 컬럼 메타데이터

## 파이프라인에서 발견된 주요 이슈

### 1. 네트워크 단절 (치명적)
- 콜백 URL이 `http://backend:8000` (Docker 내부 호스트명)
- n8n이 외부 서버(168.107.58.30)에 있으면 콜백 불가
- 결과: AI가 규칙을 생성해도 결과가 프론트에 전달되지 않음

### 2. 에러 분기 부재
- Step 8-1에서 `parse_ok=false` 경로에 연결된 노드 없음
- AI 응답 파싱 실패 시 워크플로우가 조용히 멈춤
- 사용자는 영구 "실행 중" 상태에 빠짐

### 3. 수정 후 재검토 없음
- Step 10에서 수정된 규칙이 Step 11 Merge로 바로 합쳐짐
- 수정된 SQL이 여전히 틀려도 검증 없이 최종 산출물에 포함

### 4. 대용량 처리 한계
- 50,498자 프롬프트를 한 번에 AI에 전달
- 테이블 200+ 규모 시 Gemini 컨텍스트 한계 초과 가능
- 청킹(분할 처리) 로직 없음

### 5. DBMS별 SQL 미분화
- 모든 규칙이 Oracle SQL 문법 기준 (TO_DATE, DBMS_LOB 등)
- PostgreSQL, MySQL 등 다른 DBMS에서는 실행 불가
- DBMS명 필드가 있지만 SQL 생성 시 분기하지 않음
