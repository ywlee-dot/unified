## 개선안 1: TMP/설문 테이블 필터링

### 적용 위치
Step 2 (데이터 전처리) 에서 테이블 그루핑 직후

### 왜 필요한가
식당에 43개 메뉴가 있는데, 4개는 "임시메뉴(TMP)"이고 2개는 "고객 설문지"다.
요리사(AI)에게 43개 전부 레시피를 요청하면, 임시메뉴와 설문지에도 쓸데없이 시간을 쓴다.

현재 데이터에서:
- TMP20221222, TMP20231211, TMP20240402, TMP20240418 → 이름부터 날짜가 붙은 임시/마이그레이션용 잔여 테이블
- ETCDM320 (89컬럼), ETCDM330 (55컬럼) → 전부 "강좌의 영상 시간이 적절하다고 생각하십니까?" 같은 설문 문항. 데이터 품질 규칙을 만들 의미 없음
- 이 6개 테이블이 전체 컬럼의 26% (770개 중 206개)를 차지

### 필터링 규칙
```
제외 조건:
1. 테이블명이 "TMP"로 시작 → 임시 테이블
2. 컬럼 수 > 50이면서, 컬럼명에 업무키(BIZ_CD, SBJT_NO 등)가 없음 → 설문/로그성 테이블
3. 모든 컬럼이 VARCHAR2이고 날짜/숫자 컬럼 0개 → 규칙 생성 불가 테이블
```

### 구현 방법 (n8n Step 2 코드에 추가)
```javascript
// 테이블 필터링 (Step 2 코드 끝부분에 추가)
const filteredTables = {};
for (const [tblName, tblData] of Object.entries(tables)) {
  // 1. TMP 테이블 제외
  if (tblName.startsWith('TMP')) {
    console.log(`필터링: ${tblName} (임시 테이블)`);
    continue;
  }
  
  // 2. 설문 테이블 제외 (컬럼 50개 이상 + 업무키 없음)
  const cols = tblData.columns;
  const hasBizKey = cols.some(c => 
    /BIZ_CD|SBJT_NO|FILE_NO/.test(c.영문컬럼명) && c.PK === 'Y'
  );
  if (cols.length > 50 && !hasBizKey) {
    console.log(`필터링: ${tblName} (설문/로그성 테이블, ${cols.length}컬럼)`);
    continue;
  }
  
  filteredTables[tblName] = tblData;
}
```

### 기대 효과
- 프롬프트 크기: 50,498자 → 약 30,000자 (40% 감소)
- AI 대상 테이블: 43개 → 37개
- AI가 의미 있는 테이블에 규칙을 집중 배분 가능
- Gemini 토큰 사용량 감소 → 비용 절감

## 개선안 2: SQL 컬럼 존재 자동 검증

### 적용 위치
Step 4 (생성된 규칙 파싱) 직후, Step 5 (검토) 이전에 신규 검증 단계 추가

### 왜 필요한가
AI가 만든 SQL에 "A310_BIRTH_DATE"라는 컬럼을 썼는데, 실제 테이블에는 "A310_RSCHR_BRTH"이다.
현재는 이걸 또 다른 AI(검토 Agent)가 잡아내야 하는데, AI도 놓칠 수 있다.

시험 답안지를 학생(AI)이 채점하는 격.
정답표(실제 컬럼 목록)가 있으니 컴퓨터가 채점하면 100% 정확하다.

### 검증 로직
```
1. 분석SQL에서 테이블의 컬럼명 패턴 추출
2. 해당 테이블의 실제 컬럼 목록과 대조
3. 없는 컬럼 사용 시 → 자동 "부적합" + 정확한 사유
```

### 구현 방법 (n8n Step 4 코드 뒤에 추가)
```javascript
// SQL 컬럼 존재 검증 (Step 4 이후 추가)
function validateSqlColumns(rule, tableColumns) {
  const sql = rule['분석SQL'];
  const tableName = rule['테이블명'];
  
  // 테이블의 실제 컬럼명 목록
  const actualCols = tableColumns.map(c => c.영문컬럼명.toUpperCase());
  
  // SQL에서 테이블 접두사가 붙은 컬럼명 추출
  // 예: ACDM310의 컬럼은 A310_ 접두사
  const prefix = tableName.substring(0, 4); // 대부분 4자리 접두사
  const sqlUpper = sql.toUpperCase();
  
  // 정규식으로 컬럼명 후보 추출 (영문자+숫자+_로 구성된 토큰)
  const tokens = sqlUpper.match(/[A-Z][A-Z0-9_]{3,}/g) || [];
  
  const invalidCols = [];
  for (const token of tokens) {
    // SQL 키워드 제외
    const keywords = ['SELECT', 'FROM', 'WHERE', 'AND', 'OR', 'NOT', 'NULL',
                      'COUNT', 'TRUNC', 'TO_DATE', 'TO_CHAR', 'DBMS_LOB',
                      'GETLENGTH', 'KSPS', 'VARCHAR2', 'NUMBER', 'DATE',
                      'YYYYMMDD', 'IS'];
    if (keywords.includes(token)) continue;
    if (token === tableName.toUpperCase()) continue;
    
    // 테이블 컬럼 접두사와 매칭되는 토큰만 검증
    if (token.includes('_') && !actualCols.includes(token)) {
      // 해당 테이블의 컬럼처럼 보이는데 목록에 없음
      invalidCols.push(token);
    }
  }
  
  if (invalidCols.length > 0) {
    return {
      valid: false,
      reason: `SQL에 존재하지 않는 컬럼 사용: ${invalidCols.join(', ')}`
    };
  }
  return { valid: true };
}
```

### 기대 효과
- AI 환각(hallucination)으로 만들어진 가짜 컬럼 참조 100% 감지
- AI 검토 전에 명확한 오류를 코드로 사전 제거
- AI 검토 부담 감소 → 검토 Agent가 더 깊은 논리 검증에 집중 가능

### 난이도: 중간 (코드 30줄)

## 개선안 3: 테이블별 "규칙 후보 힌트" 사전 분석

### 적용 위치
Step 2 (전처리) 이후 ~ Step 3 (AI 생성) 이전, 프롬프트에 힌트 추가

### 왜 필요한가
의사에게 환자 100명의 혈액검사 결과를 한꺼번에 주면, 가장 흔한 빈혈만 진단하고 희귀 질환은 놓친다.
대신 "이 환자는 간수치 이상 의심, 저 환자는 혈당 이상 의심"이라고 사전 분류해서 주면 진단이 정확해진다.

현재 AI에게 43개 테이블의 컬럼을 통째로 던지면, 가장 눈에 띄는 MOD_DTTM/REG_DTTM 패턴만 16개(55%) 반복한다.

### 감지 가능한 패턴

```
패턴 1: 날짜 쌍 (REG_DTTM/MOD_DTTM)
  → 감지: 같은 테이블에 _REG_DTTM과 _MOD_DTTM이 모두 존재
  → 힌트: "시간순서 일관성 후보 (수정일 >= 등록일)"
  → 거의 모든 테이블에 해당 → AI에게 "이 패턴은 3개까지만" 제한

패턴 2: 시작/종료 쌍 (STT_DATE/END_DATE, STT/END, FROM/TO)
  → 감지: 컬럼명에 STT+END 또는 START+END 쌍이 존재
  → 힌트: "시간순서 일관성 후보 (시작 <= 종료)"
  → 예: H100_NEW_STT_DATE / H100_NEW_END_DATE

패턴 3: 자기참조 FK
  → 감지: FK가 같은 테이블을 참조 (CIDC011.C011_UPR_CMN_CD → CIDC011.C011_CMN_CD)
  → 힌트: "논리관계 일관성 후보 (순환참조 검사)"

패턴 4: 상위/하위 코드 (UPR_ 접두사)
  → 감지: 컬럼명에 UPR_ 또는 _UPR이 포함
  → 힌트: "논리관계 일관성 후보 (상위코드 != 자기코드)"
  → 예: H180_UPR_BIZ_CD vs H180_BIZ_CD

패턴 5: YN 플래그 + 연관 컬럼
  → 감지: _YN으로 끝나는 컬럼 + 같은 접두사의 _DTTM 또는 _NM 컬럼
  → 힌트: "논리관계 일관성 후보 (Y이면 연관 값 존재해야)"
  → 예: C310_PDF_AUTO_SYS_CNVT_YN + C310_PDF_AUTO_SYS_CNVT_DTTM

패턴 6: 상태코드 + 조건부 필드
  → 감지: _STATUS_CD 또는 _DVS_CD 컬럼 존재
  → 힌트: "논리관계 일관성 후보 (상태값에 따른 조건부 필수)"
  → 예: H100_DCL_STATUS_CD = '2' → H100_RE_CNTN 존재 필요

패턴 7: 건수/금액/비율 숫자 컬럼
  → 감지: 한글컬럼명에 "건수/금액/비율/수량/횟수" 포함 + NUMBER 타입
  → 힌트: "계산식 후보 (>= 0)"
  → 예: M141D_DB_CNST_CNT (DB구축건수), H100_INQ_NO (조회수)

패턴 8: 월/일 범위 숫자
  → 감지: 한글컬럼명에 "월/일" 포함 + NUMBER 타입
  → 힌트: "계산식 후보 (월: 1~12, 일: 1~31)"
  → 예: H400_MM (사업월)
```

### 구현 방법 (Step 2와 Step 3 사이에 추가)
```javascript
// 테이블별 힌트 생성
function analyzeTableHints(tblName, columns) {
  const hints = [];
  const colNames = columns.map(c => c.영문컬럼명);
  
  // 패턴 1: REG/MOD 날짜 쌍
  const regDttm = colNames.find(c => c.includes('_REG_DTTM'));
  const modDttm = colNames.find(c => c.includes('_MOD_DTTM'));
  if (regDttm && modDttm) {
    hints.push(`시간순서: ${modDttm} >= ${regDttm} (단, 이 패턴은 이미 흔하므로 다른 규칙 우선)`);
  }
  
  // 패턴 2: 시작/종료 쌍
  const sttCols = colNames.filter(c => c.includes('_STT_') || c.includes('_START'));
  for (const stt of sttCols) {
    const end = colNames.find(c => c.replace('STT', 'END') === stt.replace('STT', 'END'));
    if (end) hints.push(`시간순서: ${stt} <= ${end}`);
  }
  
  // 패턴 3: 자기참조 FK
  for (const col of columns) {
    if (col.FK && col.FK.startsWith(tblName + '.')) {
      hints.push(`논리관계: ${col.영문컬럼명} 자기참조 FK → 순환참조 검사`);
    }
  }
  
  // 패턴 4: UPR_ 상위코드
  const uprCols = colNames.filter(c => c.includes('_UPR_'));
  for (const upr of uprCols) {
    const base = upr.replace('_UPR_', '_');
    if (colNames.includes(base)) {
      hints.push(`논리관계: ${upr} != ${base} (순환참조 방지)`);
    }
  }
  
  // 패턴 5: YN + 연관 DTTM
  const ynCols = colNames.filter(c => c.endsWith('_YN'));
  for (const yn of ynCols) {
    const prefix = yn.replace('_YN', '');
    const related = colNames.find(c => c.startsWith(prefix) && c !== yn);
    if (related) {
      hints.push(`논리관계: ${yn}='Y'이면 ${related} 존재 필요`);
    }
  }
  
  // 패턴 6: 상태코드
  const statusCols = colNames.filter(c => c.includes('_STATUS_') || c.includes('_STS_'));
  for (const sc of statusCols) {
    hints.push(`논리관계: ${sc} 값에 따른 조건부 필수 검사 가능`);
  }
  
  // 패턴 7: 건수/금액 숫자
  for (const col of columns) {
    if (col.데이터타입 === 'NUMBER') {
      const han = col.한글컬럼명 || '';
      if (/건수|금액|비율|수량|횟수|조회수/.test(han)) {
        hints.push(`계산식: ${col.영문컬럼명}(${han}) >= 0`);
      }
    }
  }
  
  // 패턴 8: 월/일 범위
  for (const col of columns) {
    if (col.데이터타입 === 'NUMBER') {
      const han = col.한글컬럼명 || '';
      if (/^사업월$|^월$/.test(han) || col.영문컬럼명.endsWith('_MM')) {
        hints.push(`계산식: ${col.영문컬럼명}(${han}) 범위 1~12`);
      }
    }
  }
  
  return hints;
}
```

## 개선안 4: 데이터타입-SQL 연산 호환성 체크

### 적용 위치
Step 4 (파싱) 직후, SQL 컬럼 검증과 함께 실행

### 왜 필요한가
AI가 만든 SQL이 문법적으로는 그럴듯하지만, 실제 Oracle DB에서 실행하면 에러가 날 수 있다.

번역기가 "이 약을 아침에 드세요"를 영어로 잘 번역했는데, 실제로는 저녁에 먹어야 하는 약.
문법은 맞지만 내용이 틀린 것.

### 체크 규칙

```
체크 1: VARCHAR2 날짜 vs DATE 직접 비교
  대상: 컬럼명에 _DATE, _DT가 포함 + VARCHAR2 타입
  조건: DATE 타입 컬럼과 비교하면서 TO_DATE 함수 없음
  예시:
    ❌ WHERE C024_UPD_DATE < C024_REG_DTTM
       → C024_UPD_DATE = VARCHAR2, C024_REG_DTTM = DATE
    ✅ WHERE TO_DATE(C024_UPD_DATE, 'YYYYMMDD') < TRUNC(C024_REG_DTTM)

체크 2: CLOB에 = 연산자 사용
  대상: 데이터타입이 CLOB인 컬럼
  조건: 분석SQL에서 해당 컬럼에 =, <>, LIKE 사용
  예시:
    ❌ WHERE H100_RE_CNTN = ''
    ✅ WHERE DBMS_LOB.GETLENGTH(H100_RE_CNTN) = 0

체크 3: VARCHAR2끼리 날짜 비교
  대상: 두 VARCHAR2 컬럼을 날짜로 비교
  조건: TO_DATE 없이 직접 비교 (문자열 비교는 가능하지만 경고)
  예시:
    ⚠️ WHERE H100_NEW_STT_DATE > H100_NEW_END_DATE
       → 문자열 비교라 'YYYYMMDD' 형식이면 OK, 'YYYY-MM-DD'면 OK
       → 하지만 'MM/DD/YYYY'면 틀림. 경고 수준으로 표시

체크 4: DBMS별 함수 호환
  Oracle: NVL, TO_DATE, SYSDATE, DBMS_LOB, TRUNC
  PostgreSQL: COALESCE, TO_DATE, NOW(), LENGTH, DATE_TRUNC
  MySQL: IFNULL, STR_TO_DATE, NOW(), LENGTH, DATE
```

### 구현 방법
```javascript
// 데이터타입 호환성 검증
function validateSqlTypeCompat(rule, tableColumns) {
  const sql = rule['분석SQL'];
  const warnings = [];
  
  for (const col of tableColumns) {
    const colName = col.영문컬럼명;
    if (!sql.includes(colName)) continue;
    
    const dtype = (col.데이터타입 || '').toUpperCase();
    
    // CLOB + 직접 비교
    if (dtype === 'CLOB') {
      if (sql.includes(`${colName} =`) || sql.includes(`${colName} <>`) || sql.includes(`${colName} LIKE`)) {
        if (!sql.includes('DBMS_LOB')) {
          warnings.push(`${colName}(CLOB)에 직접 비교 연산 사용. DBMS_LOB.GETLENGTH() 사용 필요`);
        }
      }
    }
    
    // VARCHAR2 날짜 + DATE 비교
    if (dtype === 'VARCHAR2' && /DATE|_DT$|_DTTM$/.test(colName)) {
      if (!sql.includes(`TO_DATE(${colName}`) && !sql.includes(`TO_CHAR(`)) {
        // 같은 SQL에 DATE 타입 컬럼도 있는지 확인
        const hasDateCol = tableColumns.some(c => 
          c.데이터타입 === 'DATE' && sql.includes(c.영문컬럼명)
        );
        if (hasDateCol) {
          warnings.push(`${colName}(VARCHAR2)을 DATE 타입 컬럼과 비교 시 TO_DATE 변환 필요`);
        }
      }
    }
  }
  
  return warnings;
}
```

### 기대 효과
- 생성된 SQL을 실제 Oracle DB에 바로 실행 가능
- 타입 불일치로 인한 ORA-01722(invalid number), ORA-01861(literal does not match) 에러 사전 방지
- 현재는 담당자가 수동으로 SQL을 검증해야 하는데, 이 과정을 자동화

### 난이도: 중간 (코드 40줄)

## 개선안 5: 규칙 다양성 강제 검증

### 적용 위치
Step 11 (Merge) 직후, Step 11-1 (형식 매핑) 이전

### 왜 필요한가
시험문제 30개를 만들었는데 20개가 덧셈, 5개가 뺄셈, 5개가 곱셈. 나눗셈은 0개.
이러면 시험지 역할을 못한다.

현재 결과: 29개 중 16개(55%)가 "수정일 >= 등록일" 동일 패턴.
공공데이터 품질 평가에서 "다양한 테이블에 걸쳐 규칙이 정의되어 있는가"가 평가 항목이므로, 커버리지와 다양성이 중요하다.

### 검증 규칙

```
체크 1: 동일 패턴 편중
  방법: 분석SQL 패턴을 정규화하여 그루핑
  기준: 동일 패턴이 전체의 40% 초과 → 경고
  예시:
    "WHERE {X}_MOD_DTTM < {X}_REG_DTTM" 패턴 → 16개(55%) → 경고
    → 이 중 10개 삭제하고, 해당 테이블에서 다른 유형 규칙 재생성 요청

체크 2: 품질지표 비율
  기준: 3개 지표 중 하나가 60% 초과 → 경고
  현재: 시간순서 55% → 통과하지만 경계
  목표: 각 지표 25~40%

체크 3: 테이블 커버리지
  기준: 필터링 후 테이블 중 커버율 70% 미만 → 경고
  현재: 18/43 = 42% → 경고
  조치: 미커버 테이블 목록을 AI에 전달하여 추가 생성 요청
  
체크 4: 테이블당 규칙 수 편중
  기준: 한 테이블에 4개 이상 → 경고
  현재: HPDM100에 5개 → 경고
  조치: 초과분 중 MOD>=REG 같은 흔한 패턴 삭제, 다른 테이블로 배분
```

### 구현 방법
```javascript
// 다양성 검증 (Step 11 이후)
function validateDiversity(rules, allTables) {
  const issues = [];
  
  // 체크 1: 패턴 편중
  const patterns = {};
  for (const r of rules) {
    // SQL을 정규화 (테이블명, 컬럼 접두사 제거)
    const normalized = r['분석SQL']
      .replace(/KSPS\.\w+/g, 'TABLE')
      .replace(/\w{1,5}_/g, 'X_');
    patterns[normalized] = (patterns[normalized] || 0) + 1;
  }
  for (const [pat, count] of Object.entries(patterns)) {
    const ratio = count / rules.length;
    if (ratio > 0.4) {
      issues.push({
        type: '패턴편중',
        message: `동일 패턴 ${count}개(${(ratio*100).toFixed(0)}%): ${pat.substring(0, 60)}...`,
        severity: 'warning'
      });
    }
  }
  
  // 체크 2: 품질지표 비율
  const indicators = {};
  for (const r of rules) {
    const ind = r['품질지표명'];
    indicators[ind] = (indicators[ind] || 0) + 1;
  }
  for (const [ind, count] of Object.entries(indicators)) {
    if (count / rules.length > 0.6) {
      issues.push({
        type: '지표편중',
        message: `${ind}: ${count}개(${(count/rules.length*100).toFixed(0)}%)`,
        severity: 'warning'
      });
    }
  }
  
  // 체크 3: 테이블 커버리지
  const coveredTables = new Set(rules.map(r => r['테이블명']));
  const uncovered = allTables.filter(t => !coveredTables.has(t));
  const coverRate = coveredTables.size / allTables.length;
  if (coverRate < 0.7) {
    issues.push({
      type: '커버리지부족',
      message: `${coveredTables.size}/${allTables.length} (${(coverRate*100).toFixed(0)}%)`,
      uncoveredTables: uncovered,
      severity: 'warning'
    });
  }
  
  // 체크 4: 테이블당 편중
  const perTable = {};
  for (const r of rules) {
    perTable[r['테이블명']] = (perTable[r['테이블명']] || 0) + 1;
  }
  for (const [tbl, count] of Object.entries(perTable)) {
    if (count >= 4) {
      issues.push({
        type: '테이블편중',
        message: `${tbl}: ${count}개 규칙 (최대 2개 권장)`,
        severity: 'info'
      });
    }
  }
  
  return issues;
}
```

### 검증 실패 시 대응 방안

```
방안 A: 자동 재생성 (루프)
  다양성 검증 실패 → 미커버 테이블 목록 + "MOD>=REG 패턴 제외" 지시를 추가하여 AI에 재요청
  → 추가 생성된 규칙을 기존에 Merge
  → 최대 2회 반복 후 확정
  
방안 B: 리포트 첨부 (수동 보완)
  다양성 검증 결과를 엑셀에 별도 시트로 첨부
  → 담당자가 미커버 테이블에 대해 수동으로 규칙 추가
  → 현실적으로 더 안전한 방안
```

- 테이블 커버리지 42% → 80%+
- 품질지표 균형 배분
